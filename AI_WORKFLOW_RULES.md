# AI_WORKFLOW_RULES.md

## تعليمات إلزامية قبل تنفيذ أي مهمة

قبل تنفيذ أي تعديل أو إنشاء أو حذف أو اقتراح أي كود داخل هذا المشروع، يجب عليك الالتزام بالتعليمات التالية بدقة:

---

# 1) قراءة ملفات التوثيق أولًا

يجب قراءة جميع ملفات التوثيق الموجودة داخل مجلد `Docs` وفهمها بالكامل قبل تنفيذ أي عملية.

الملفات الأساسية التي يجب مراجعتها دائمًا:

- `Docs/PROJECT_CONTEXT.md`
- `Docs/ARCHITECTURE.md`
- `Docs/DATABASE.md`
- `Docs/API.md`
- `Docs/BUSINESS_RULES.md`
- `Docs/SECURITY.md`
- `Docs/TASKS.md`
- `Docs/SETUP_GUIDE.md`

ممنوع تنفيذ أي تعديل قبل فهم:

- بنية المشروع.
- العلاقات بين المكونات.
- قواعد العمل.
- قاعدة البيانات.
- أسلوب كتابة الكود الحالي.

---

# 2) تحليل المشروع قبل التعديل

قبل كتابة أي كود يجب عليك:

- تحليل الملفات المرتبطة بالمهمة.
- فهم تدفق البيانات.
- فهم تأثير التعديل على باقي النظام.
- اكتشاف أي Dependencies مرتبطة.
- مراجعة Naming Convention المستخدمة بالمشروع.
- الالتزام بنفس أسلوب البرمجة الحالي.

---

# 3) تحديث ملفات التوثيق تلقائيًا

بعد أي تعديل داخل المشروع يجب تحديث ملفات `Docs` تلقائيًا.

أمثلة:

- عند إضافة جدول → تحديث `DATABASE.md`
- عند إضافة API → تحديث `API.md`
- عند تعديل Business Logic → تحديث `BUSINESS_RULES.md`
- عند إضافة ميزة → تحديث `PROJECT_CONTEXT.md`
- عند تعديل الهيكل → تحديث `ARCHITECTURE.md`

---

# 4) إنشاء سجل تغييرات دائم

بعد كل عملية تعديل يجب إنشاء أو تحديث ملف:

`Docs/CHANGELOG.md`

ويحتوي على:

- تاريخ التعديل.
- الملفات المعدلة.
- وصف التغييرات.
- سبب التعديل.
- التأثير المتوقع.
- أي مشاكل محتملة.
- هل تم تحديث التوثيق أم لا.

صيغة المثال:

```md
## 2026-06-07

### Added

- إضافة نظام تسجيل المواعيد.

### Modified

- تعديل API إنشاء الحجز.
- تحديث جدول appointments.

### Files Changed

- controllers/AppointmentController.php
- models/Appointment.php
- routes/api.php

### Documentation Updated

Yes
```

# 5) التعليقات داخل الكود

أي كود جديد يجب أن يحتوي على:

- Comments بالعربي.
- شرح واضح للدوال.
- شرح المتغيرات المهمة.
- شرح أي Logic معقد.

ممنوع كتابة كود بدون توثيق واضح.

---

# 6) الحفاظ على استقرار المشروع

ممنوع:

- حذف أي كود بدون تحليل تأثيره.
- كسر التوافق مع الأجزاء الحالية.
- تغيير بنية قاعدة البيانات بدون تحديث التوثيق.
- إنشاء أكواد مكررة.
- تجاهل Security أو Validation.

---

# 7) قبل تسليم أي تعديل

يجب عليك دائمًا:

- شرح ما تم تعديله.
- ذكر الملفات التي تغيرت.
- ذكر الملفات التي تم تحديثها داخل Docs.
- ذكر أي تأثير على قاعدة البيانات أو الـ API.
- اقتراح تحسينات إن وجدت.

---

# 8) أسلوب العمل المطلوب

اعمل كمحلل نظم + مهندس برمجيات + موثق تقني في نفس الوقت.

أي تعديل يجب أن يكون:

- منظم.
- قابل للصيانة.
- موثق بالكامل.
- متوافق مع المشروع الحالي.
- سهل الفهم لأي مطور أو AI لاحقًا.

---

# 9) الأولوية دائمًا للتوثيق الحقيقي

لا تقم بكتابة توثيق افتراضي أو تخميني.

يجب استخراج المعلومات مباشرة من:

- الكود الحالي.
- قاعدة البيانات.
- الملفات الموجودة.
- العلاقات الحقيقية داخل المشروع.

---

# 10) قواعد مهمة جدًا

- إذا كانت المعلومات غير واضحة قم بتحليل المشروع أولًا قبل الإجابة.
- إذا كان هناك تعارض بين الكود والتوثيق قم بتصحيح التوثيق.
- إذا وجدت مشاكل معمارية أو أمنية قم بتوثيقها.
- حافظ على Markdown منظم واحترافي.
- لا تتجاهل تحديث ملفات Docs بعد أي تعديل.
