# 🛡️ الجوانب الأمنية (Security)

تحليل وتوثيق الجوانب الأمنية الخاصة بمنصة "طبيبي". يتناول هذا الملف آليات الحماية المطبقة في النظام، وكشف نقاط الضعف المعمارية التي تحتاج إلى ترقيع.

## 🔐 1. آلية التحقق والصلاحيات (Authentication & Authorization)
- **إدارة الجلسات عبر Tokens**: لا يستخدم النظام الجلسات التقليدية (PHP Sessions/Cookies) لتسجيل الدخول، بل يعتمد على نظام `Bearer Token` يُحفظ في الـ `LocalStorage` للمتصفح (وهذا ضروري لعمل الـ API مع تطبيقات الجوال React Capacitor).
- **التحقق من الصلاحيات (RBAC)**: 
  يتم فحص أدوار المستخدمين بدقة عبر طبقة الـ `AuthMiddleware.php`. يحتوي هذا الكلاس على دوال تمنع الوصول غير المصرح به:
  - `AuthMiddleware::patientOnly()` للـ Endpoints الخاصة بالمرضى فقط.
  - `AuthMiddleware::authenticate()` للأطباء والعيادات.
  - أي خلل أو Token منتهي الصلاحية يقوم برفض الطلب وإرسال حالة `401 Unauthorized`.

## 🛡️ 2. إجراءات الحماية المطبقة حالياً
1. **منع حقن قواعد البيانات (SQL Injection Protection)**:
   يستخدم كلاس النواة (`Database.php`) الـ `PDO` مع `Prepared Statements` (كتابة المتغيرات كـ `?` وتمريرها لاحقاً). هذا يمنع كلياً عمليات حقن أو تغيير هيكل الاستعلامات الخبيثة.
2. **التحقق من المدخلات (Input Validation)**:
   قبل تنفيذ أوامر الحفظ (مثل الحجز)، تقوم الـ Controllers بالتحقق من المدخلات الأساسية وتنظيفها، وإرجاع `success: false` إذا كانت المدخلات فارغة أو بصيغة خاطئة.

## ⚠️ 3. نقاط الضعف المحتملة (Vulnerabilities) واقتراحات التحسين

### أ. ضعف تشفير كلمات المرور (Base64)
- **المشكلة**: تخزين كلمات المرور كنصوص مُشفرة بـ `Base64` في جدول `users`. هذا يعتبر ترميزاً وليس تشفيراً أحادي الاتجاه (Hashing)، مما يعني أن أي تسريب لقاعدة البيانات سيكشف كلمات السر الحقيقية للمستخدمين فورا!
- **السبب**: مطابقة كلمات المرور مع خوارزميات قديمة في تطبيق Delphi المكتبي للعيادات.
- **الحل الواجب تطبيقه**:
  1. تحديث برنامج Delphi لكي يستخدم `Bcrypt`.
  2. تشغيل سكريبت في الـ Backend يعيد تشفير كل كلمات المرور باستخدام الدالة الأصلية في PHP: `password_hash($pwd, PASSWORD_BCRYPT)`.
  3. تغيير التحقق ليصبح باستخدام `password_verify()`.

### ب. تخزين التوكن في LocalStorage
- **المشكلة**: تخزين الـ Token في LocalStorage يعرضه لهجمات Cross-Site Scripting (XSS).
- **الحل**: لا بأس بذلك إذا تم فحص وتنظيف كل المُدخلات من المستخدمين بدقة (تعقيم مدخلات الدردشة والتقييمات)، ولكن يُفضل للويب تخزين التوكن في `HttpOnly Secure Cookie`.

### ج. إعدادات الـ CORS
- **المشكلة**: مسار الـ API قد يكون مُعدّاً بقيمة `Access-Control-Allow-Origin: *` في ملف `.htaccess`، مما يسمح لأي موقع بالاتصال بالـ API.
- **الحل**: يجب تحديد الدومين الفعلي فقط في خادم الإنتاج:
  `Access-Control-Allow-Origin: https://tabibi.dz`

### د. حماية الـ API من الهجمات المكثفة (Rate Limiting)
- لا يوجد حالياً نظام `Rate Limiter` (تقييد عدد الطلبات) في الـ PHP.
- **الحل**: ضبط إعدادات الـ Nginx / Apache في الخادم (Production) لتحديد عدد الطلبات في الثانية لمنع هجمات الـ DDoS والـ Brute Force على شاشة الـ Login.
